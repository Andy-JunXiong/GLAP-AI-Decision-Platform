Decision logic

Readable translation of the deployed Lambda rules

glap-ai-agent-orchestrator · prod v2
01 · ADMIT INPUT

Select pending anomalies

Read anomaly_flag = 1, exclude any anomaly that already has a decision, rank by newest date and absolute z-score, then cap each run at 10.

NOT EXISTS decision
ORDER BY run_date DESC,
abs(z_score) DESC
LIMIT 10
02 · EXPLAIN

Generate root cause

Transit-duration increases receive 0.80 confidence; SLA breach increases receive 0.75; |z| ≥ 2 receives 0.65; other early deviations receive 0.55.

avg_leg_duration_days ↑ → 0.80
breach_rate ↑ → 0.75
|z_score| ≥ 2 → 0.65
else → 0.55
03 · DECIDE

Assign action and priority

High-confidence transit anomalies become carrier/port investigations. High-confidence breach-rate anomalies become SLA escalations. Other cases remain monitored.

duration + confidence ≥ .75 → HIGH
breach + confidence ≥ .70 → HIGH
otherwise → MEDIUM monitor
04 · IDEMPOTENCY

Prevent duplicate outputs

Before every INSERT, check run_date + entity_key + metric_name. Existing root causes can be reused if an earlier run failed before writing the decision.

duplicate key =
run_date + entity_key + metric_name
05 · SAFE SQL

Escape generated values

Every string written into generated SQL doubles single quotes. Numeric conversion falls back safely instead of breaking orchestration.

value.replace("'", "''")
to_float(value, default=0.0)
06 · DRY RUN

Preview without writes

Candidate and staging smoke tests read pending anomalies and generate decision previews, but do not INSERT root-cause or decision records.

{"dry_run": true}
→ previews[]
→ no records written

SQL workbench

The actual operational query patterns and what OPS uses them for

Athena · curated_iceberg
01 · Pending anomaly intakeREAD · production orchestrator
SELECT
  anomaly.run_date,
  anomaly.entity_key,
  anomaly.metric_name,
  anomaly.metric_value,
  anomaly.baseline_value,
  anomaly.z_score
FROM curated_iceberg.fact_ai_anomaly_scores_v1 anomaly
WHERE anomaly.anomaly_flag = 1
  AND NOT EXISTS (
    SELECT 1
    FROM curated_iceberg.fact_ai_decision_explanations_v1 decision
    WHERE decision.run_date = anomaly.run_date
      AND decision.entity_key = anomaly.entity_key
      AND decision.metric_name = anomaly.metric_name
  )
ORDER BY anomaly.run_date DESC, abs(anomaly.z_score) DESC
LIMIT 10;
OPS question

Which abnormal entities still need a decision?

Control

NOT EXISTS prevents already-decided anomalies from re-entering the queue.

Priority

Newest and most statistically extreme anomalies are handled first.

Capacity

LIMIT 10 bounds runtime and avoids unbounded daily work.

02 · Idempotency check before INSERTREAD · write guard
SELECT COUNT(*) AS cnt
FROM <target_table>
WHERE run_date = DATE '<run_date>'
  AND entity_key = '<entity_key>'
  AND metric_name = '<metric_name>';
OPS question

Did this run already create the same root cause or decision?

Recovery behavior

If root cause exists but decision does not, the next run reuses the root cause and resumes the missing decision.

Run evidence

Lambda returns inserted and skipped counters so operators can distinguish new work from safe reprocessing.

03 · Decision volume and priority validationREAD · operational verification
SELECT action_priority,
       COUNT(*) AS decision_count
FROM curated_iceberg.fact_ai_decision_explanations_v1
GROUP BY action_priority
ORDER BY decision_count DESC;

SELECT run_date,
       COUNT(*) AS decision_count
FROM curated_iceberg.fact_ai_decision_explanations_v1
GROUP BY run_date
ORDER BY run_date DESC
LIMIT 10;

Is the pipeline creating plausible daily decision volume and severity distribution?

Watch for

Zero decisions despite anomalies, sudden HIGH spikes, or repeated historical dates.
